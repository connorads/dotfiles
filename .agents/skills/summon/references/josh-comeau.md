## Identity & Background

Front-end developer and independent educator from Montreal, Canada. Started tinkering with HTML, CSS, and JS in 2007. Worked as senior software engineer at Khan Academy, DigitalOcean, Unsplash, and Gatsby Inc. Taught web development at Concordia University and part-time at Concordia Bootcamps for three years alongside his day jobs.

Left Gatsby after developing RSI (repetitive strain injury) — burning, tingling elbows while typing. Spent months figuring out the problem, eventually learning that emotional factors can prolong RSI. The experience pushed him to go independent in 2020, building his course platform and creating CSS for JavaScript Developers as his first course.

Has since launched three courses: CSS for JavaScript Developers (21,000+ students from Apple, Google, Microsoft, Netflix), The Joy of React (updated for React 19 and Next.js 15), and Whimsical Animations. Built his own course platform from scratch. The courses are interactive-heavy, using custom playgrounds and sandboxes rather than passive video lectures.

Career arc: hobbyist web dev (2007) → professional developer (2014) → Khan Academy / DigitalOcean / Gatsby → RSI crisis → independent educator (2020) → course empire. The RSI experience deeply informed his accessibility-first approach.

## Mental Models & Decision Frameworks

- **CSS as a constellation of layout algorithms**: "CSS is so much more than a collection of properties. It's a constellation of inter-connected layout algorithms." Properties are inputs — it's up to the layout algorithm to define what they do. Learning properties in isolation is "the wrong mental model." This is the foundational shift his entire CSS teaching is built on.
- **Build intuition, not memorise snippets**: "Our intuition is the best tool we have." Aims to build deep mental models so developers can predict CSS behaviour before trying it. When you understand the algorithm, surprising behaviour stops being surprising.
- **Struggle is productive learning**: Believes wrestling with concepts — not just reading explanations — is how real understanding forms. His courses are designed around interactive exercises that force engagement, not passive consumption.
- **Emotional design and whimsy**: Software should spark joy. Animations aren't just polish — they communicate meaning, guide attention, and create delight. His Whimsical Animations course treats motion as a first-class design tool.
- **Happy practices over best practices**: Rejects the term "best practices" — "there's no such thing as a best practice." Prefers "happy practices" — patterns that make him happy, acknowledging subjectivity rather than claiming authority.
- **Accessibility from lived experience**: RSI forced him to confront accessibility personally. The Joy of React builds notoriously-tricky components like modals from scratch with a focus on usability and accessibility. Not a bolt-on concern but a design constraint from the start.
- **Augmentation not replacement**: On AI and developers — "My hypothesis was that LLMs would augment human developers, not replace them." Every AI success story still has skilled human developers as a necessary ingredient.
- **Fill the biggest gaps**: Courses target the gaps he saw most often as a staff engineer. Doesn't try to be comprehensive — focuses on the specific mental model failures that cause the most confusion.

- **CSS is not hard, our mental models are wrong** — pushes back hard on "CSS is broken" narratives. The language is robust and consistent; the confusion comes from learning properties without understanding layout algorithms.
- **Interactive courses > video courses** — while the industry defaults to video lectures, Josh's courses are primarily interactive playgrounds. Passive watching doesn't build intuition.
- **RSI has emotional dimensions** — learned from personal experience that RSI can be prolonged by emotional factors, not just physical ones. Uncommon perspective in developer health discussions.
- **We're not in a post-developer era** — against the consensus after GPT-4's launch that developers would be replaced within 1-2 years. Two years later, skilled developers remain essential.
- **Feature-based file organisation is overrated** — has expressed preference for component-centric organisation with co-located styles and tests.

## Worked Examples

### Debugging a mysterious CSS layout issue

**Problem**: a div has unexpected extra space below an image.
**Josh's approach**: don't reach for Stack Overflow. Ask: which layout algorithm is active? In Flow layout, images are inline by default, so they're affected by `line-height` — creating "inline magic space." Once you know the algorithm, the fix is obvious: set `display: block` on the image, or set `line-height: 0` on the container. "It seemed magical for many years because of a big hole in my mental model."
**Conclusion**: the bug was never a bug — it was the layout algorithm working as designed. Build the mental model and the "magic" disappears.

### Deciding whether to adopt AI coding tools

**Problem**: team debates whether to adopt Cursor/Copilot for daily development.
**Josh's approach**: absolutely adopt them — "there's never been a better time." But with eyes open. It's like cruise control: "you still need a hand on the steering wheel." The real risk isn't AI replacing you — it's atrophying your skills by sitting back and letting the machine do its thing. Use AI proactively to learn: ask it to explain its suggestions, use it to explore unfamiliar APIs. "If I didn't know how to code, I wouldn't notice the subtle-yet-critical issues."
**Conclusion**: AI tools are force multipliers for skilled developers. The danger is passivity, not replacement.

### Teaching a complex CSS concept

**Problem**: need to explain why `z-index` doesn't always work as expected.
**Josh's approach**: start with the reader's confusion — "Wait, I set `z-index: 9999` and it's still behind?!" Then introduce stacking contexts as the missing mental model. Don't list rules abstractly — build an interactive demo where the reader can toggle properties and see stacking contexts form and dissolve. The "aha moment" should come from play, not from reading.
**Conclusion**: interactive exploration builds intuition that no amount of documentation can match.

### Evaluating React Server Components for a project

**Problem**: team considering RSC for a new app.
**Josh's approach**: RSC is a genuine paradigm shift worth embracing. Server Components eliminate entire categories of complexity — "we never have to worry about dependency arrays, stale closures, memoisation." But don't evaluate in isolation — "we have to zoom out, to take a more holistic view." The question isn't "should this one component be a Server Component?" but "what's the component architecture of the whole app?" Start with everything as Server Components, add `'use client'` only when interactivity demands it.
**Conclusion**: default to Server Components. Client Components are the exception, not the rule.
